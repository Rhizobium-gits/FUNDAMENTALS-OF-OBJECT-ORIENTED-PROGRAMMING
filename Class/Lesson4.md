コンセプト
**ポリモーフィズム（多態性）**は、オブジェクト指向プログラミングにおいて、同じメソッド呼び出しに対して、それぞれのオブジェクトが自分の都合に合わせて異なる振る舞いをする概念である [引用][引用]。これは、アルファベット池で生まれたパンクの仏教的な「有意識」の考え方に例えられ、同じ現象に対してそれぞれのエージェントが勝手に異なる意味を解釈して仕事をするというリアルワールドの概念に基づいている [引用]。
カプセル化は、ポリモーフィズムと密接に関係しており、機械の内部動作を隠蔽し、使い方さえわかればよいという考え方でソフトウェアを扱う設計方針である [引用]。クラスを作る側は、使う側に対して使い方さえわかればよく、中身はわからなくてよいように意図的に隠す（隠蔽）という概念である [引用]。
デザインパターンは、オブジェクト指向でコードを書いてきた人々のナレッジとして蓄積された、クラスとクラスをうまく組み合わせる「オブジェクト指向あるある」のパターン集である [引用]。代表的なものとして、テンプレートメソッドパターン（複数のクラスに似たような処理をさせつつ、具体的な部分はサブクラスで実装することを明示する） [引用]や、アダプターパターン（異なるクラスを同じインターフェースで扱えるようにする） [引用][引用]がある。
継承の仕組みでは、親クラスから子クラス、孫クラスへと拡張していく際、差分プログラミングという考え方で変えたいところだけを変える [引用]。データ（変数）の追加、メソッドの追加、メソッドの上書き（オーバーライド）が可能であり、親クラスで定義されたメソッドもsuperキーワードで呼び出すことができる [引用][引用]。
理解したこと
ポリモーフィズムの基本動作原理
ポリモーフィズムの核心は、親クラスの代理として子クラスが働けるという代用可能性（サブスティティナビリティ）にある [引用][引用]。スポットクラスの代わりにアンパンマンやバイキンマンがいて、それぞれのアップデートメソッドを動かすと、それぞれが独自の方法で動いて表示される [引用]。
重要なのは、メソッドは実行時のオブジェクトの型に基づいて呼ばれるという点である [引用][引用]。親クラスのインスタンス変数として定義していても、実際にインスタンス化されたのが子クラスであれば、子クラスのメソッドが実行される（ダイナミックローディング） [引用]。これにより、柔軟なコードが書けるようになる [引用][引用]。
継承における変数とメソッドの違い
授業では、変数とメソッドでは上書きの振る舞いが異なるという重要な点が強調された [引用][引用]。
メソッドの場合：
- 孫クラスのインスタンスで子クラスのメソッドPを実行すると、その中で呼ばれるメソッドFは孫クラスで定義されたものが実行される [引用][引用][引用]
- (親)孫インスタンス.F()のようにキャストしても、実際のオブジェクトが孫なので孫のメソッドが実行される [引用][引用][引用]
- super.F()と明示的に書くことで、スーパークラスのメソッドを呼び出せる [引用][引用]
変数の場合：
- this.Xは省略されており、実際には定義されたクラスの変数が呼ばれる [引用][引用]
- (親)this.Xと書くと、親クラスで定義されたXが呼ばれる [引用][引用]
- 変数は上書きされているように見えて、実はパラレルに存在している [引用][引用][引用]
具体例として、親クラスでX=0、子クラスでX=1、孫クラスでX=2と定義した場合、子クラスのメソッドPから呼び出すと、Xは1、(親)this.Xは0、super.Xも0となる [引用][引用][引用]。これは変数が動的にオーバーライドされず、定義された場所の変数が呼び出されるためである [引用][引用]。
クラス設計の実践的アプローチ
スケッチという概念は、Processing特有の考え方で、最終的な作品になる手前の下絵やアイデアを書き止めるものを指す [引用][引用][引用]。これは日本語で言う「習作」に相当し、完成品として作っているのではなく、何かを考えるために作っているようなイメージである [引用][引用]。
実際の開発プロセスとして、ちょっとずつ部品を作っては確認し、部品と部品を組み合わせる確認をするという単体テストと統合テストの繰り返しが推奨される [引用]。部品Aと部品Bがうまく組み合わさったら、次に部品Cを単体で作成し、また組み合わせ方を確認していくという流れである [引用]。
シンセシス3のサンプルでは、RPGゲームの要素を段階的に構築する例が示された [引用][引用][引用]：
1. シンセシス1：矢印キーで円を移動させる基本機能（30行程度） [引用]
2. シンセシス2：部屋とダガー（剣）、ゲートを描画 [引用]
3. シンセシス3：鍵のアイテムを追加 [引用][引用]
4. シンセシス4：ルーム切り替え機能と衝突判定 [引用][引用]
5. シンセシス5：画像を使った完成版 [引用][引用]
ピンポンゲームの構造分解
シンセシス3のもう一つの例として、ピンポンゲームの段階的構築が紹介された [引用][引用][引用]：
- エクササイズ1：マウス上下でパドルが動く [引用]

エクササイズ4は93行だったが、エクササイズ5では44行+ボールクラス59行となり、単純な行数は増えている [引用]。しかし、名詞でまとまっているがゆえに新しい名詞を追加しやすくなっているという利点がある [引用][引用]。
オブジェクト指向における名詞の重要性
オブジェクト指向は、世界を名詞で切り分ける方法である [引用]。アルゴリズムが動詞的であるのに対し、オブジェクト指向は人間が物理世界を認識している方法（名詞による認識）に近い形でソフトウェアを書く [引用][引用]。
関数の名前付けの例として、drawI()という名前をつけた瞬間に2つ書こうかなと思うように、名前の付け方がその次のスケッチや拡張の方向性を導く [引用][引用]。例えば、drawCircles()という名前にすると、グラデーションで変わる円を追加したくなるなど、名前が発想を誘導する [引用][引用]。
ボールとパドルという名詞を違う名詞に変えると違うゲームに見えるという点も重要である [引用]。ブロック崩しにしようとブロックという名詞を追加すると、ボールとパドルという名前がふさわしくなくなる [引用][引用]。クラス名を疑ってみて、別のものに見えたらどうなるかを妄想しながらリファクタリングするのが、講師の発想法の一つである [引用][引用]。
完全には理解できなかったこと
メモリ構造における変数の重複保持
変数の上書きが実はパラレルに存在しているという概念は、メモリ構造の理解が必要である。質問で確認された内容によると、孫クラスのインスタンスを生成すると、親クラスのメモリ領域と子クラスのメモリ領域が肥大化して建て増しされる形になる [引用][引用][引用]。
古い木造の家に鉄筋のビルを建て増ししたような構造で、同じ名前の変数Xが実は複数の場所に存在している [引用][引用]。this.X、(親)this.X、super.Xという異なるアクセス方法で、それぞれ異なる場所のXにアクセスできる [引用][引用]。
ただし、変数を無意識にオーバーライドすることは推奨されない [引用][引用]。全く違う人が作ったコードのサブクラスを定義した際、スーパークラスのスーパークラスあたりで定義されていた変数と同じ名前を無意識に定義してしまい、変な振る舞いになるという実例が紹介された [引用][引用]。
instanceofによる型チェック
instanceof演算子は、実際にインスタンス化されたオブジェクトが何のクラスのインスタンスなのかを調べる機能である [引用][引用]。親クラスの変数に孫クラスのインスタンスを代入した場合、親インスタンス instanceof 親も親インスタンス instanceof 孫も両方trueになる [引用]。
これは、拡張されているので完全に親のフリができるためである [引用]。しかし、実際にどのサブクラスのインスタンスなのかを調べないと、どのメソッドがどう実行されるかが確定しないため、instanceofでの確認が必要になる場合がある [引用][引用]。
キャストの限界
(子供)孫インスタンス.P()のように子供クラスにキャストしても、実際のオブジェクトが孫クラスなので孫クラスのメソッドが実行される [引用][引用]。これは、変数に実際入っているオブジェクトの型に基づいてメソッドが呼ばれるという原則による [引用]。
キャストは「子供クラスになったつもりで子供なんですって言いながら」メソッドを実行するが、本人は孫クラスのインスタンスなので孫のメソッドが実行される [引用]。この動作は直感的でない場合があり、混乱の原因となる [引用][引用]。
デバッグとテストの重要性
徹夜してコードを書くと絶対にバグが入るため、若くてもやらない方がよいとされる [引用]。また、AIと一緒に作業するとAIの仕事が早いため休憩しにくく、朝からAIを使うと3時頃には脳が限界になることがある [引用][引用]。気分転換や休憩は重要であり、特にAI時代においても人間の物理的な身体は疲れるため、適切な休息が必要である [引用][引用]。
問題の切り分けも重要で、うまくいかない場合はうまくいっているところまで戻って、追加しては確認するというプロセスが推奨される [引用]。部品は部品で確認し、組み合わせがうまくいくかを確認し、違う組み合わせで問題が起きたら切り分けるという手法である [引用]。

ポリモーフィズムの核心原則
- メソッドは実行時のオブジェクトの型に基づいて呼ばれる（ダイナミックローディング） [引用][引用][引用]
- 親クラスの変数に子クラスのインスタンスを代入しても、子クラスのメソッドが実行される [引用][引用]
- super.メソッド名()で意図的にスーパークラスのメソッドを呼び出せる [引用][引用]
- 代用可能性（サブスティティナビリティ）：親の代わりに子供に同じ仕事をお願いできる [引用][引用]
変数とメソッドの上書きの違い
- 変数：定義された場所の変数が呼ばれる（動的にオーバーライドされない） [引用][引用][引用]
- メソッド：実行時のインスタンスのメソッドが呼ばれる（動的にオーバーライドされる） [引用][引用]
- this.変数名は省略形で、実際には定義されたクラスの変数 [引用]
- (親)this.変数名で親クラスの変数にアクセス可能 [引用][引用]
- 変数は上書きされているように見えて、実はパラレルに存在している [引用][引用]
デザインパターンの種類
- テンプレートメソッドパターン：アブストラクトクラスで共通処理を定義し、具体的な部分はサブクラスで実装 [引用]
- アダプターパターン：異なるクラスを同じインターフェースで扱えるようにする。サブクラス定義と委譲（部品として持つ）の2つの実装方法がある [引用][引用]
- シングルトンパターン、オブザーバーパターン、ステートパターンなども存在 [引用][引用]
継承の仕組み
- 差分プログラミング：変えたいところだけ変える [引用]
- 親クラスの変数とメソッドは子クラスでそのまま使える [引用]
- メソッドは上書き（オーバーライド）可能 [引用][引用]
- 上書きしたメソッドでも、superで親クラスのメソッドを呼び出せる [引用]
- 新しいデータ（変数）やメソッドを追加できる [引用][引用]
オブジェクト指向の設計原則
- カプセル化：使い方さえわかればよく、中身は隠蔽する [引用]
- 名詞で切り分ける：人間が物理世界を認識する方法に近い [引用][引用]
- スケッチ（習作）：最終作品の前の下絵、アイデアを書き止めるもの [引用][引用][引用]
- 単体テストと統合テスト：部品を作っては確認し、組み合わせを確認する [引用]
実装上の注意点

- キャストしても実際のオブジェクトの型に基づいてメソッドが呼ばれる [引用][引用]
- 変数の無意識なオーバーライドは避けるべき（混乱の原因） [引用][引用]
- デバッグ時は出力を表示して変数を確認する [引用]
- 問題を切り分けてうまくいっているところまで戻る [引用]
